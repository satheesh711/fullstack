16.what is d/f b/w block elements vs inline elements?

Block:
Always starts on a new line before and after the element itself.
It occupy the full width of the webpage or entire horizontal space within the their container.
It have default margin top and bottom values to create space between them and other elements.
Inline:
Always starts on same line with other inline elements.
It occupies the width according to their content.
Inline elements do not have default value of margin top and bottom.

17.what is semantic vs non-semantic elements?

Semantic Elements:
Semantic elements clearly describe their meaning in a way that both the browser and developers can understand. These elements define the structure and content of a webpage, making it more readable and accessible, especially for search engines and screen readers.
Examples:
header: Defines a header section of a webpage.
nav: Represents a navigation section.
article: Represents an independent piece of content.
section: Defines a section in a document.
footer: Represents the footer of a page.
Purpose: They improve code readability and SEO by making the structure of the page clearer.
Non-Semantic Elements:
Non-semantic elements do not provide any meaning about their content. They are generic containers, typically used for styling and layout purposes.
Examples:
div: A generic block-level container.
span: A generic inline container.
Purpose: They don't describe the content and are often used for styling or grouping elements when no specific meaning is needed.

18. d/f b/w div and span?

Differences Between div and span:
Purpose:
div: Block-level element used for grouping larger content or sections.
span: Inline element used for styling small portions of text.
Display Behavior:
div: Starts on a new line and takes up the full width (causes line breaks).
span: Stays inline with surrounding text and only takes as much width as needed.
Usage:
div: Ideal for larger layouts and sections.
span: Best for applying styles to a small segment within text.

19.table vs li ?

Differences Between table and li:
Purpose:
table: Displays tabular data in rows and columns.
li: Represents an individual item in a list (ordered or unordered).
Structure:
table: Requires tr, th, and td for structure.
li: Used within ul or ol without additional structure.
Display Behavior:
table: Shows data in a grid format.
li: Displays items as a list, usually with bullets or numbers.
Use Cases:
table: Best for complex data presentations.
li: Ideal for simple lists or menus.

20.what are meta tags?

Meta tags are HTML elements in the section that provide metadata about a web page. They help search engines and

Advantages:

27.what re the features of html 5?

HTML5 introduces several features and enhancements that improve web development and user experience. Here are the key features:
New Semantic Elements: Introduces elements like header, footer, article, section, and nav for better document structure and SEO.
Multimedia Support: Native support for audio and video with audio and video elements, eliminating the need for plugins like Flash.
Canvas Element: Provides the canvas element for drawing graphics and animations via JavaScript, enabling dynamic visual content.
Form Enhancements: New input types (e.g., email, date, range) and attributes (e.g., placeholder, required) improve form functionality and validation.
Local Storage: Allows web applications to store data locally in the browser using localStorage and sessionStorage, enabling offline access and better performance.
Geolocation API: Provides a standardized way to access a user's geographical location, useful for location-based services.
Improved Accessibility: Enhancements for better accessibility features, including ARIA roles and attributes.
Responsive Design Support: New attributes and elements (like picture and srcset) facilitate responsive web design for various devices and screen sizes.

29.what is git???

Git is a distributed version control system used to track changes in source code during software development.
Key Points:
Version Control: Keeps a history of changes and allows reverting to previous versions.
Branching and Merging: Enables independent work on features or fixes and combines changes seamlessly.
Distributed: Each developer has a complete copy of the repository, allowing offline work.
Collaboration: Facilitates teamwork while managing code conflicts.
Efficiency: Fast operations for committing changes and merging branches.
Open Source: Free to use and supported by a large community.

30.what is d/f b/w git vs github?

Git and GitHub are related but serve different purposes:
Git
Definition: A distributed version control system for tracking changes in source code.
Functionality: Manages code versions locally; allows branching, merging, and offline work.
Usage: Operated through the command line or GUIs. GitHub
Definition: A cloud-based platform that hosts Git repositories.
Functionality: Facilitates collaboration, sharing, and project management among developers.
Features: Offers issue tracking, pull requests, and other collaboration tools; requires internet access.

31.what is local repo and github repo?

A local repository and a GitHub repository both store your project files but in different locations:
Local Repository
Definition: A Git repository stored on your computer or local machine.
Purpose: Used to track changes and manage versions of your project locally, without needing an internet connection.
Usage: You can clone repositories, make commits, and create branches on your local system. However, to share your changes with others, you need to push them to a remote repository (e.g., on GitHub).
GitHub Repository
Definition: A Git repository hosted on GitHub's cloud platform.
Purpose: Used for sharing your code, collaborating with others, and managing your project online.
Usage: Once changes are made in your local repository, they can be pushed to GitHub for others to access, collaborate on, and review.

38.what is the flow of git???

Initialize or Clone: Start a new repository or clone an existing one.
Create a Branch (optional): Make a new branch for feature development.
Make Changes: Edit files in your local repository.
Stage Changes: Add modified files to the staging area.
Commit Changes: Save your changes with a descriptive message.
Push Changes: Upload your commits to the remote repository.
Pull Changes: Fetch and integrate updates from the remote repository.
Merge Changes: Combine your feature branch into the main branch when complete.
Resolve Conflicts: Fix any merge conflicts that arise.
Repeat: Continue the workflow for ongoing development.
